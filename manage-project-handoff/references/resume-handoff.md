# 恢復交接

在新 session 要理解、檢查或延續既有交接時執行本流程。

## 1. 依優先序讀取

先讀目前有效的系統、使用者與專案指示，再依序讀取：

1. 適用的 `AGENTS.md`。
2. `docs/private/HANDOFF.md`。
3. 交接引用的 `TODO.md`、`docs/PRD.md`、README、決策文件與關鍵程式碼。
4. 與下一步直接相關的設定與測試。

交接不存在時，不要自行建立假快照。改以 PRD、TODO、Git 與儲存庫內容重建可驗證狀態，並說明缺少前一個 session 的臨時脈絡。

恢復模式不得只因 `docs/private/` 或忽略規則缺失就建立資料夾或修改 `.gitignore`；只有使用者要求建立新交接時，才切換至建立流程補建。

## 2. 驗證新鮮度

檢查目前 Git root、branch、HEAD、staged／unstaged／untracked、進行中的 Git 操作及交接引用路徑，並比較快照：

- branch 或 HEAD 不同時列出差異，不推定交接仍適用。
- 工作樹與快照不符時，以目前檔案為準並標示快照過時。
- 已完成、阻塞或驗證敘述沒有目前證據時，降級為待確認。
- 使用者要求與目前 `AGENTS.md`、PRD、TODO 或新指示衝突時，採目前明確指示並回報衝突；無法判定時停止請使用者決定。
- 不因交接寫著「測試通過」就沿用結果；是否重跑依接續工作、狀態變化與風險決定。

不得為了匹配舊快照而 reset、checkout、清理或覆寫目前工作。

## 3. 恢復使用者意圖

將交接中的要求分成：

- 仍有效且範圍明確。
- 需要使用者確認是否延續。
- 已被目前指示或正式來源取代。
- 與現況衝突。

只把第一類當成目前限制。長期規範候選尚未寫入 `AGENTS.md` 時，不得擴大到原範圍之外。措辭模糊或沒有失效條件時，採最窄合理範圍並標示假設。

## 4. 回報恢復結果

開始任何實作前，簡潔回報：

1. 目前目標與對應 PRD／TODO。
2. 仍有效的使用者要求及適用範圍。
3. 已驗證的 branch、HEAD、工作樹與測試狀態。
4. 交接與現況的差異、過時內容及衝突。
5. 一至三個可信下一步。
6. 必須先由使用者決定的阻塞。

若使用者只要求讀取交接或說明目前狀態，到此停止。

## 5. 延續工作

只有使用者同時要求繼續工作時才執行下一步：

- 依任務使用適合的實作、文件或 Git Skill。
- 保留已確認限制，並在範圍可能擴大前重新確認。
- 只重跑因 HEAD、工作樹、環境或時間變化而失效的驗證。
- 工作狀態實質改變後，先同步 TODO／README 等正式來源；再次切換 session 時再執行建立交接流程。

不得在恢復時自動刪除或清空舊快照。專案完成或交接不再需要時，提出刪除建議並等待確認。
