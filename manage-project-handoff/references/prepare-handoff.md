# 建立交接

在使用者要求建立、更新或準備 session 交接時執行本流程。

## 1. 確認範圍與來源

先讀取目前有效的 `AGENTS.md`、既有 `docs/private/HANDOFF.md`、`TODO.md`、`docs/PRD.md`、README，以及與目前工作直接相關的程式碼與文件。檢查 Git root、branch、HEAD、staged／unstaged／untracked、進行中的 Git 操作與本次已執行的驗證。

只使用目前對話、儲存庫、工具結果與既有文件支持的事實。無法驗證的舊交接內容標示為過時或待確認，不得沿用成現況。

## 2. 分流使用者要求

逐項辨識明確要求：

1. 已在正式來源中：只引用來源與必要位置。
2. 本輪或特定任務限定：保留在交接快照，記錄範圍與失效條件。
3. 可能長期適用：列為 `AGENTS.md` 候選並等待使用者確認；確認前仍以交接傳遞，不直接修改 `AGENTS.md`。
4. 產品需求或實作狀態：依主 Skill 分流至 PRD 或 TODO。
5. 與既有規範衝突：並列內容、來源與影響，停止自動升級或合併。

不要把助理自行採用的工作方式偽裝成使用者要求。只從上下文推測的要求標示「推測，待確認」或省略。

## 3. 同步正式來源

- 實作進度、下一步或阻塞已改變時，使用 `$write-project-todo` 更新本機 TODO。
- 已驗證的公開行為、指令、設定或限制已改變時，使用 `$write-project-readme` 同步雙語 README。
- 產品需求確實改變且使用者要求更新時，使用 `$write-project-prd`。
- 長期規範只有取得明確確認後才更新 `AGENTS.md`。

其他 Skill 修改檔案後，重新檢查 Git 狀態及受影響事實。交接本身不得觸發程式碼實作。

## 4. 建立私有邊界

1. 使用既有私有文件慣例；沒有時使用 `docs/private/HANDOFF.md`。
2. 增量建立 `docs/private/`，並確認根目錄 `.gitignore` 含獨立規則 `docs/private/`。
3. 檢查目標是否已追蹤或曾存在於提交歷史。已追蹤時停止，回報取消追蹤與歷史風險，不自行修改 index。
4. 寫入後以 `git check-ignore -v docs/private/HANDOFF.md` 驗證。
5. 若接手者不共用目前工作區，先確認使用者指定的安全傳遞方式；沒有安全管道時只回報限制，不公開或暫存交接檔。

發現憑證或高度敏感值時不要複製或摘要內容；只記錄風險類型與撤銷、輪替或移出專案的必要處置。

## 5. 更新單一快照

增量更新而非附加新 session 區塊：

- 保留仍正確的限制、決策、阻塞與下一步。
- 以目前證據更新 branch、HEAD、工作樹與驗證狀態。
- 移除明確失效或被取代的短期內容；原因仍影響後續決策時保留精簡理由。
- 將大量細節改成相對路徑、task ID 或 commit 引用。
- 下一步限制為一至三項可直接開始且按依賴排序的動作。

時間使用可驗證的目前時間與時區。不要虛構 session ID、完成日期、測試結果或負責人。

## 6. 驗證與回報

確認快照可讓未參與前一個 session 的讀者理解目標、限制、狀態與起點；所有路徑存在或標示尚未建立；Git 與驗證結果符合現況；使用者要求包含範圍、狀態、來源與期限；快照被忽略且未 staged／tracked；沒有機密、完整 diff、完整 PRD／TODO 或冗長對話。

最後回報路徑、同步的正式來源、忽略驗證、長期規範候選、待確認事項與下一個 session 的建議讀取順序。
