# 管理專案交接

以單一私有快照補足 PRD 與 TODO 無法表達的即時脈絡。交接文件是暫時狀態索引，不是第二份需求文件、TODO、永久規範或逐次累積的 session log。

## 核心原則

- 先讀取目前有效的指示與儲存庫事實；交接內容不得凌駕目前使用者要求或專案規範。
- 只記錄下一個 session 接手確實需要、可驗證且仍有效的內容，不複製完整 PRD、TODO、README、diff 或對話。
- 使用者要求必須保留原意、適用範圍、狀態與有效期限；不得自行推定為永久規範。
- 快照預設放在 `docs/private/HANDOFF.md`，並以 `docs/private/` 排除 Git；優先重用 `$initialize-project-structure` 預建的私有邊界。
- 若私有邊界因 clone、舊專案或既有結構而缺失，只有建立模式可依本 Skill 相容補建；恢復模式只回報缺失，不修改結構。
- `docs/private/` 只降低誤提交風險，不是機密保管庫；不得寫入 token、密碼、私鑰或完整敏感值。
- 被忽略的本機快照只會留在目前工作區；跨機器或開發者交接時，必須由使用者指定安全傳遞方式，不得改為上傳公開 Git。
- 建立交接不代表授權執行下一步、修改正式規範、提交、推送、刪除或清理工作樹。

## 模式與必要 Reference

| 模式 | 使用時機 | 必須完整讀取 |
|---|---|---|
| 建立交接 | 建立、整理、更新或準備 session 交接 | [`references/prepare-handoff.md`](references/prepare-handoff.md) |
| 恢復交接 | 新 session 要理解、檢查或延續既有交接 | [`references/resume-handoff.md`](references/resume-handoff.md) |

若同時要求先讀取再更新，依序完整讀取兩份 reference。不要載入未使用模式的 reference。

## 資訊分流

| 資訊 | 正式位置 | 交接文件處理 |
|---|---|---|
| 長期且跨 session／任務適用的專案規範 | `AGENTS.md` | 記為長期規範候選；只有使用者明確要求或確認後才移入 |
| 產品問題、目標、範圍、需求與驗收條件 | `docs/PRD.md` | 只引用路徑與必要 ID，不複製全文 |
| 任務、進度、相依、阻塞與實作下一步 | `TODO.md` | 只引用相關任務；必要時先使用 `$write-project-todo` 同步 |
| 已驗證且適合公開的功能、指令與限制 | `README.md`、`README.en.md` | 必要時先使用 `$write-project-readme` 同步 |
| 本輪限定要求、待確認事項、未提交狀態、測試結果與短期決策 | `docs/private/HANDOFF.md` | 保存足以接手的精簡快照 |
| 憑證或高度敏感內容 | 專用機密管理 | 不寫入；只記錄不揭露值的風險與必要處置 |

產品需求確實改變且使用者要求同步時才使用 `$write-project-prd`。`$manage-project-docs` 不應將 `HANDOFF.md` 公開化；整理 `docs/` 時，交接快照預設維持私有。

## 使用者要求紀錄

每項需要跨 session 傳遞的要求至少記錄：

- **要求**：精簡但不改變原意；措辭重要時可保留必要短句。
- **適用範圍**：整個專案、特定功能、檔案、任務或本輪工作。
- **狀態**：已確認、本輪有效、長期規範候選、待確認或已被取代。
- **來源**：例如「使用者明確要求」；不得虛構日期、人物或決策權。
- **期限或失效條件**：未知時明確標示未指定。
- **衝突與後續**：列出與現有規範的差異，以及下一個 session 應採取或避免的行為。

已存在於 `AGENTS.md`、PRD、TODO 或 README 的內容只建立引用。遇到衝突時保留雙方並請使用者決定，不自行合併或推定優先序。

## 快照最低內容

只加入有可靠內容的章節：

1. 更新時間與時區、專案根目錄、branch、HEAD。
2. 目前目標與範圍。
3. 使用者要求與限制。
4. 已完成工作與證據。
5. 工作樹、未提交檔案及目前 Git 狀態。
6. 已執行與未執行的驗證及結果。
7. 重要決策、理由及仍相關的捨棄方案。
8. 阻塞、風險與待確認事項。
9. 依序排列的一至三個下一步。
10. 必讀文件與關鍵檔案的相對路徑。

不得以無證據的「全部完成」或「測試通過」填補缺項。沒有已知阻塞時可寫「未發現已知阻塞」，不得推定不存在風險。

## 隱私與 Git 邊界

- 建立交接前先檢查 `docs/private/` 與 `.gitignore` 的獨立規則 `docs/private/`；兩者已正確時直接重用，不得產生無意義變更。
- 若規則存在但資料夾缺失，建立模式可在寫入快照前建立資料夾；若規則缺失，先保留既有內容並增量補上。不得重複規則或建立 `docs/private/.gitkeep`。
- 使用 `git check-ignore` 驗證快照被忽略。若檔案已追蹤，先停止並回報；`.gitignore` 不會取消追蹤或清除歷史。
- 只有使用者明確要求公開交接內容時，才評估建立去除內部脈絡的公開文件；不得直接追蹤 `HANDOFF.md`。
- 刪除或清空未追蹤的交接快照前取得確認，因為 Git 通常無法復原。

## 完成回報

建立模式列出快照路徑、資訊來源、同步的正式文件、私有邊界是重用或補建、Git 忽略狀態、保留的使用者要求、待確認與未驗證事項。恢復模式列出已恢復的目標、有效限制、與現況差異、可信下一步及阻塞。

除非使用者同時要求繼續實作，恢復交接後只回報狀態，不自動執行 TODO、修改程式碼、提交或推送。
