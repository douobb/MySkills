# 管理專案文件

整理 `docs/` 內真正有長期價值的內容，避免將暫存紀錄或內部資料當成公開文件。流程分為「唯讀盤點」與「確認後執行」兩階段；即使使用者一開始要求直接整理，也必須先完成盤點並等待確認。

## 核心原則

- 先遵循專案既有文件慣例、導覽、命名及文件工具設定；不要另建第二套架構。
- 根據完整內容、讀者、正確性、引用關係與長期價值分類，不得只看檔名、日期或篇幅。
- 保留能解釋架構決策、限制、取捨與原因的獨特內容；程式碼通常只能說明「做了什麼」，不一定說明「為什麼」。
- 不建立空白分類目錄、模板文件或為了形式完整而補出的內容。文件很少時可留在 `docs/` 根目錄。
- 將「公開適合度」與「內容品質」分開判斷；整理完成不代表適合公開。
- `docs/private/` 只降低意外提交風險，不是機密保管庫。不得把 token、密碼、私鑰或其他憑證移入其中當作保護措施。
- 只處理 `docs/`；除非使用者明確擴大範圍，不整理根目錄 README、Issue、Wiki 或原始碼內文件。

## 第一階段：唯讀盤點

### 1. 建立文件清單

先檢查：

- `AGENTS.md`、README、`.gitignore` 與文件工具設定。
- `docs/` 內所有一般檔案與隱藏檔案，包括 Markdown、文字、設定、圖片、PDF 及其他文件格式。
- Git 追蹤與忽略狀態；可用時檢查 `git status`、`git ls-files` 與 `git check-ignore`。
- README、程式碼、設定及其他文件對目標檔案的連結或引用。
- 文件之間的重複、衝突、版本關係與可能的事實來源。

不得跟隨指向專案外部的符號連結。對支援的二進位文件使用適當工具讀取或渲染；無法可靠讀取時標示「待確認」，不得歸為無作用。

### 2. 使用四種分類

| 分類 | 判定方式 | 預設處置 |
|---|---|---|
| 無作用 | 純暫存 log、重複輸出、已失效且沒有獨特資訊，對使用者與後續開發均無合理價值 | 提議刪除，但先保留 |
| 需整理 | 混合有用內容與雜訊、與其他文件重疊、資訊零散，或位置與受眾不明 | 提出來源、目標檔案與整合方案 |
| 已整理公開檔案 | 內容清楚、正確、適合目標讀者，不含不應公開資訊，且可作為公開文件維護 | 保留或移至合適的公開位置 |
| 已整理不公開檔案 | 內容已有用且完整，但含內部產品規劃、商業考量、未公開決策或其他不宜公開資訊 | 提議移至 `docs/private/` 並排除 Git |

若文件含有任何仍具價值的獨特內容，不得歸為「無作用」；應歸為「需整理」。無法確認正確性、用途或敏感度時，說明不確定性並請使用者判定。

### 3. 回報並停止

依四個分類條列所有檔案，每個檔案至少回報：

- 相對路徑。
- 一至兩句內容簡介；敏感文件只提供不揭露細節的高階摘要。
- 分類與判定理由。
- Git 已追蹤、未追蹤或已忽略狀態。
- 建議動作、目標位置及主要風險。

另列出：

- 建議刪除的精確檔案清單。
- 每個整合方案的來源檔、目標檔、預計保留內容與來源檔後續處置。
- 建議的最小 `docs/` 架構，只列這次實際需要的目錄。
- 建議新增或修正的 `.gitignore` 規則。
- 需要使用者判定的衝突、敏感度或文件真實性問題。

回報後停止。未取得使用者對精確刪除、整合、移動及 Git 變更方案的明確確認前，不得建立目錄、修改 `.gitignore`、移動、改寫或刪除任何檔案。部分確認只授權已明確同意的項目。

## 第二階段：確認後執行

### 1. 重新確認現況

執行前重新檢查已核准檔案是否被修改、移動或刪除。若盤點結果已過時、發現新衝突或範圍改變，停止並更新方案，不要套用舊結論。

### 2. 整理公開文件

既有慣例優先；沒有慣例時，依實際內容選用下列位置：

```text
docs/
├── tutorials/              # 引導新手完成學習歷程
├── guides/                 # 完成具體工作的操作指南
├── reference/              # API、設定、Schema 與準確事實
├── explanation/            # 背景、概念、原理與取捨
├── architecture/           # 系統架構與設計
│   └── decisions/          # ADR 或重要決策紀錄
├── operations/             # 部署、維運、故障排除與 Runbook
└── assets/                 # 文件實際引用的圖片或附件
```

只建立至少有一個核准文件要放入的目錄。小型專案不必拆成多層；參考文件的結構應盡量反映它描述的系統結構。只有確實改善導覽時才建立或更新 `docs/README.md` 或索引，不得產生空索引。

### 3. 整理不公開文件

- 將核准的不公開整理文件放入 `docs/private/`，並在專案根目錄 `.gitignore` 保留獨立規則 `docs/private/`。
- 不要在 `docs/private/` 建立 `.gitkeep`，因為該目錄本身不應被提交。
- 保留既有 `.gitignore` 內容，只增量加入缺少的規則，不得重複。
- 若 `docs/PRD.md` 是既有 Skill、腳本或流程的固定來源，可保留原路徑並維持獨立忽略規則；除非使用者同意同步修改所有引用，否則不要逕自搬移。
- `.gitignore` 不影響已追蹤檔案。若私密文件已被 Git 追蹤，先回報精確清單；只有使用者明確同意取消追蹤時才調整索引。
- 若內容曾推送到遠端，說明加入 `.gitignore` 不會清除 Git 歷史。不得自動重寫歷史、強制推送或聲稱資料已從遠端移除。
- 若發現真實憑證或高度敏感資料，停止搬移與摘要，避免輸出值，並建議撤銷或輪替；不要把 `docs/private/` 描述為安全補救。

### 4. 整合與清理

對每個核准方案：

1. 先建立或更新目標文件，保留所有仍正確且獨特的資訊、決策理由、限制與必要引用。
2. 去除重複、過時紀錄、對話殘片與無法支持的結論；不要補寫未驗證事實。
3. 統一目標讀者、術語、標題、檔名與相對連結。
4. 驗證目標文件完整且可取代來源文件。
5. 更新受影響的 README、索引、設定與交叉連結。
6. 最後才處理使用者核准的來源檔。只刪除精確核准的檔案；未追蹤檔刪除前須提醒可能無法由 Git 復原，並在可行時採可復原方式。

不得因整理而修改產品需求、程式行為或技術決策。遇到內容衝突時保留差異並回報，不要自行裁決。

## 完成驗證與回報

完成前確認：

- 所有原始文件都有最終分類或明確待確認狀態。
- 所有刪除、整合、移動與 Git 變更都在核准範圍內。
- 公開文件不含已識別的私密規劃、憑證或不必要內部紀錄。
- `docs/private/` 已被忽略，且已追蹤檔案的例外已正確揭露。
- 沒有建立空分類目錄、重複文件或空白索引。
- 相對連結、圖片、文件工具導覽及主要引用沒有因搬移而失效。
- `git status` 只包含預期變更；未自動暫存、提交或推送。

最後列出：保留、建立、整合、移動、刪除的檔案，更新的連結與 `.gitignore` 規則，仍待確認事項，以及私密文件的 Git 追蹤與遠端歷史風險。

## 設計參考

- [Diátaxis](https://diataxis.fr/start-here/)：依教學、操作指南、參考與解釋區分讀者需求；只採用實際有助益的分類。
- [Git `gitignore` 文件](https://git-scm.com/docs/gitignore)：忽略規則只影響未追蹤檔案，不能清除既有追蹤或遠端歷史。
