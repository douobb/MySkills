# 管理專案文件

整理 `docs/` 內真正有長期價值的內容，避免將暫存紀錄或內部資料當成公開文件。流程分為「唯讀盤點」與「確認後執行」兩階段；即使使用者一開始要求直接整理，也必須先完成盤點並等待確認。

## 核心原則

- 先遵循專案既有文件慣例、導覽、命名及文件工具設定；不要另建第二套架構。
- 根據完整內容、讀者、正確性、引用關係與長期價值分類，不得只看檔名、日期或篇幅。
- 保留能解釋架構決策、限制、取捨與原因的獨特內容；程式碼通常只能說明「做了什麼」，不一定說明「為什麼」。
- 不建立空白分類目錄、模板文件或為了形式完整而補出的內容。文件很少時可留在 `docs/` 根目錄。
- 將「公開適合度」與「內容品質」分開判斷；整理完成不代表適合公開。
- 公開文件必須能脫離被忽略、未追蹤或私有的 TODO、PRD 與內部規劃獨立理解，並清楚回答「誰需要閱讀」及「讀完有何用途」。
- 無法證明具有獨立公開價值時，預設歸為不公開；不要因排版正式、資訊很多或含有獨特內容就推定適合上傳。
- `docs/private/` 只降低意外提交風險，不是機密保管庫。不得把 token、密碼、私鑰或其他憑證移入其中當作保護措施。
- 將 `$initialize-project-structure` 預建的 `docs/private/` 視為標準本機私有邊界；若已存在就直接重用，不得加入 `.gitkeep` 或重複新增忽略規則。
- 只處理 `docs/`；除非使用者明確擴大範圍，不整理根目錄 README、Issue、Wiki 或原始碼內文件。

## 第一階段：唯讀盤點

### 1. 建立文件清單

先檢查：

- `AGENTS.md`、README、`.gitignore` 與文件工具設定。
- `docs/` 內所有一般檔案與隱藏檔案，包括 Markdown、文字、設定、圖片、PDF 及其他文件格式。
- Git 追蹤與忽略狀態；可用時檢查 `git status`、`git ls-files` 與 `git check-ignore`。
- README、程式碼、設定及其他文件對目標檔案的連結或引用。
- 文件之間的重複、衝突、版本關係與可能的事實來源。
- 每份文件是否依賴被忽略、未追蹤、私有或不會上傳的 TODO、PRD 與其他文件。
- 是否含有未定義的 `TASK-*`、`FR-*`、`NFR-*`、`DEC-*`、Checkpoint、Sprint、Milestone、Phase 或階段標記。
- 內容屬於已實作且可驗證的行為、明確公開的 roadmap，或尚未實作的內部契約。
- 公開文件是否能由 README、文件索引或其他公開入口找到，而不是孤立檔案。
- `docs/private/` 與根目錄 `.gitignore` 中獨立規則 `docs/private/` 是否存在；資料夾缺失可能只是空白忽略目錄未隨 clone 保留，也可能代表舊專案尚未初始化。
- 將 `docs/private/HANDOFF.md` 辨識為 `$manage-project-handoff` 管理的操作快照；可盤點並歸為已整理不公開，但預設不得提議公開化、搬移或合併。
- 將 `docs/PRD.md` 辨識為 `$write-project-prd` 的預設私有正式來源；可盤點其公開適合度，但未經使用者明確要求不得提議公開化、搬移、合併或刪除。

不得跟隨指向專案外部的符號連結。對支援的二進位文件使用適當工具讀取或渲染；無法可靠讀取時標示「待確認」，不得歸為無作用。

### 2. 使用四種分類

| 分類 | 判定方式 | 預設處置 |
|---|---|---|
| 無作用 | 純暫存 log、重複輸出、已失效且沒有獨特資訊，對使用者與後續開發均無合理價值 | 提議刪除，但先保留 |
| 需整理 | 含有可保留內容，但混合雜訊、私有追蹤標記、未驗證規劃、缺少讀者背景，或依賴不會上傳的文件 | 可行時提出「語意化公開改寫」，否則提出私有整合方案 |
| 已整理公開檔案 | 內容清楚、正確、已驗證、具明確讀者與閱讀價值，可獨立理解且通過全部公開准入條件 | 保留或移至合適的公開位置 |
| 已整理不公開檔案 | 內容已有用且完整，但價值主要來自內部追蹤、產品階段、驗收矩陣、商業考量、未公開決策或其他私有脈絡 | 提議移至 `docs/private/` 並排除 Git |

若文件含有任何仍具價值的獨特內容，不得歸為「無作用」；應依完成度與公開適合度歸為「需整理」或「已整理不公開檔案」。無法確認正確性、用途或敏感度時，說明不確定性並請使用者判定。

### 3. 套用公開准入門檻

只有同時符合下列條件，文件才可歸為「已整理公開檔案」：

1. 明確定義目標讀者、文件目的與讀完後可完成或理解的事情。
2. 不依賴未上傳的 TODO、PRD、私有文件或未公開規劃，單獨閱讀仍可理解。
3. 不含未定義的內部 TASK、FR、NFR、DEC、Checkpoint、Sprint、Milestone、Phase 或階段代號。
4. 必須保留的版本、階段或識別碼已在公開文件中定義，且對讀者確實有用。
5. 已區分可由程式碼、測試、設定或發布資料驗證的目前行為、公開限制與未來規劃；不得把未實作契約寫成現有能力。
6. 不揭露私有產品策略、商業考量、內部驗收關係或安全敏感資訊。
7. 可由 README、公開索引或其他公開文件找到，並具備必要交叉連結。
8. 對使用者或新開發者有持續維護價值，而不是只用來證明 Codex、單一任務或某一開發階段做過什麼。

任一條件不符合，就不得歸為已整理公開。若能在不洩漏私有脈絡下改寫成獨立且有用的文件，歸為「需整理」；若移除私有脈絡後會失去主要價值，歸為「已整理不公開檔案」。無法證明公開價值時，預設採後者。

### 4. 回報並停止

依四個分類條列所有檔案，每個檔案至少回報：

- 相對路徑。
- 一至兩句內容簡介；敏感文件只提供不揭露細節的高階摘要。
- 分類與判定理由，以及公開准入門檻的通過／失敗項目。
- 依賴的私有或不會上傳文件，以及未定義的內部追蹤標記。
- Git 已追蹤、未追蹤或已忽略狀態。
- 建議動作、目標位置及主要風險。

另列出：

- 建議刪除的精確檔案清單。
- 每個整合方案的來源檔、目標檔、預計保留內容與來源檔後續處置。
- 對每個「需整理」文件評估語意化公開改寫與私有整合兩種方向，說明推薦選項；若公開改寫不可行，明列失敗條件。
- 建議的最小 `docs/` 架構，只列這次實際需要的目錄。
- 建議新增或修正的 `.gitignore` 規則。
- 需要使用者判定的衝突、敏感度或文件真實性問題。

回報後停止。私有邊界缺失在此階段只列為相容性問題；未取得使用者對精確刪除、整合、移動及 Git 變更方案的明確確認前，不得建立目錄、修改 `.gitignore`、移動、改寫或刪除任何檔案。部分確認只授權已明確同意的項目。

## 第二階段：確認後執行

### 1. 重新確認現況

執行前重新檢查已核准檔案是否被修改、移動或刪除。若盤點結果已過時、發現新衝突或範圍改變，停止並更新方案，不要套用舊結論。

### 2. 整理公開文件

既有慣例優先；沒有慣例時，依實際內容選用下列位置：

```text
docs/
├── tutorials/              # 引導新手完成學習歷程
├── guides/                 # 完成具體工作的操作指南
├── reference/              # API、設定、Schema 與準確事實
├── explanation/            # 背景、概念、原理與取捨
├── architecture/           # 系統架構與設計
│   └── decisions/          # ADR 或重要決策紀錄
├── operations/             # 部署、維運、故障排除與 Runbook
└── assets/                 # 文件實際引用的圖片或附件
```

只建立至少有一個核准文件要放入的目錄。小型專案不必拆成多層；參考文件的結構應盡量反映它描述的系統結構。只有確實改善導覽時才建立或更新 `docs/README.md` 或索引，不得產生空索引。

公開整理不是機械刪除內部代號。執行公開改寫時：

1. 先定義文件的目標讀者、目的、適用範圍與目前狀態。
2. 將 `對應 TASK-003` 等追蹤語句改寫成可獨立理解的領域語意，例如「本節定義 Session 狀態轉移」，而不是只刪除代號。
3. 將內部 Phase 改成已公開且有定義的版本／能力狀態；若只是未公開規劃，移入私有文件。
4. 只描述由程式碼、測試、設定或正式發布資料支持的目前行為。尚未驗證的規格保留私有，或明確放入已公開 roadmap。
5. 從私有來源建立最小公開投影，不複製私有驗收關係、排程、商業脈絡或不必要細節。
6. 補上必要背景、術語與公開連結，並從 README 或公開索引建立入口。
7. 重跑公開准入門檻；仍有任一失敗項目就移入私有，不得勉強發布。

### 3. 整理不公開文件

- 將內部產品階段、TASK／FR 追蹤、驗收矩陣、Codex 工作證據，以及無法脫離私有 PRD／TODO 理解的完整規格預設放入私有文件。
- 優先重用初始化時既有的 `docs/private/` 與根目錄 `.gitignore` 獨立規則 `docs/private/`，不要在狀態已正確時改寫任何內容。
- 若規則存在但資料夾因 clone 或舊工作區而缺失，只在核准的不公開文件確實需要目的地時建立資料夾；不要只為補齊空架構而建立。
- 若資料夾或獨立忽略規則缺失，只有在使用者核准的方案內才增量建立或補上；保留既有 `.gitignore` 內容且不得重複規則。
- 不要在 `docs/private/` 建立 `.gitkeep`，因為該目錄本身不應被提交。
- `docs/private/HANDOFF.md` 是 `$manage-project-handoff` 的保留操作快照；除非使用者明確將它納入本次處理範圍，否則不得改寫、整合、搬移或刪除，其內容更新應交由交接 Skill。
- `docs/PRD.md` 是 `$write-project-prd` 的保留正式來源，預設維持原路徑與獨立忽略規則；除非使用者明確納入範圍並同意同步修改所有引用，否則不得改寫、公開化、整合、搬移或刪除。
- `.gitignore` 不影響已追蹤檔案。若私密文件已被 Git 追蹤，先回報精確清單；只有使用者明確同意取消追蹤時才調整索引。
- 若內容曾推送到遠端，說明加入 `.gitignore` 不會清除 Git 歷史。不得自動重寫歷史、強制推送或聲稱資料已從遠端移除。
- 若發現真實憑證或高度敏感資料，停止搬移與摘要，避免輸出值，並建議撤銷或輪替；不要把 `docs/private/` 描述為安全補救。

### 4. 整合與清理

對每個核准方案：

1. 先建立或更新目標文件，保留所有仍正確且獨特的資訊、決策理由、限制與必要引用。
2. 去除重複、過時紀錄、對話殘片與無法支持的結論；不要補寫未驗證事實。
3. 統一目標讀者、術語、標題、檔名與相對連結。
4. 驗證目標文件完整且可取代來源文件。
5. 更新受影響的 README、索引、設定與交叉連結。
6. 最後才處理使用者核准的來源檔。只刪除精確核准的檔案；未追蹤檔刪除前須提醒可能無法由 Git 復原，並在可行時採可復原方式。

不得因整理而修改產品需求、程式行為或技術決策。遇到內容衝突時保留差異並回報，不要自行裁決。

## 完成驗證與回報

完成前確認：

- 所有原始文件都有最終分類或明確待確認狀態。
- 所有刪除、整合、移動與 Git 變更都在核准範圍內。
- 每份公開文件都通過全部公開准入條件，具有明確讀者價值及公開入口。
- 公開文件不依賴被忽略、未追蹤、私有或不會上傳的文件。
- 公開文件不含已識別的私密規劃、憑證或不必要內部紀錄。
- 已掃描 TASK、FR、NFR、DEC、Checkpoint、Sprint、Milestone、Phase 與階段標記；只保留已公開定義且對讀者有用者。
- 私有邊界已按核准方案重用或修復，`docs/private/` 已被忽略、沒有重複規則，且已追蹤檔案的例外已正確揭露。
- `docs/private/HANDOFF.md` 維持私有，且未在未授權下被改寫、整合、搬移或刪除。
- `docs/PRD.md` 維持其正式來源與預設私有狀態，或其公開／搬移已取得明確授權並同步所有引用。
- 沒有建立空分類目錄、重複文件或空白索引。
- 相對連結、圖片、文件工具導覽及主要引用沒有因搬移而失效。
- `git status` 只包含預期變更；未自動暫存、提交或推送。

最後列出：保留、建立、整合、移動、刪除的檔案，更新的連結與 `.gitignore` 規則，仍待確認事項，以及私密文件的 Git 追蹤與遠端歷史風險。
