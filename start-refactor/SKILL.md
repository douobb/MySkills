# Start Refactor Skill

本 Skill 旨在協助開發者將 `code-review` skill 的審查意見轉化為具體的程式碼改進。核心目標是在保持功能不變的前提下，提升程式碼品質、安全性與可維護性。

## When to use this skill

- 當需要依據先前 Code Review 的建議進行修復時。
- 當程式碼結構變得臃腫、難以維護，需要進行系統性優化時。
- 在修復 Bug 或增加新功能前，需要先進行預防性重構時。

## How to use it

### 1. 重構心態 (Mindset)

- **安全第一**：重構不應改變既有的外部行為。
- **漸進式修改**：小步快跑 (Small Steps)。每一次修改都應該是可驗證的。
- **驗證先行**：重構前應先與使用者確認驗證場景。
- **合作驗證**：測試主要由 Agent 與使用者手動執行，無需編寫自動化測試程式碼，但需確保修改後的邏輯符合預期。
- **對接審查**：嚴格遵循 `code-review` 提供的高優先級建議。

### 2. 修復與改進原則 (Guidelines)

#### 遵循審查建議
- **對應修復**：找出 `code-review` 輸出中的「詳細說明」，逐項對應修改。
- **優先順序**：優先修復「嚴重程度：高」的問題（如安全性漏洞或邏輯錯誤）。

#### 安全重構步驟
1. **環境確認**：確保當前工作目錄乾淨。
2. **定義驗證場景**：在修改前，與使用者溝通如何手動測試此功能的正確性。
3. **原子化修改**：一次只做一件事（例如：僅修改變數命名、或僅提取函式）。
4. **手動驗證**：完成原子修改後，Agent 應主動告知使用者可以進行哪些手動檢查。
5. **漏洞修補**：整合安全性修復作為重構的一部分，而不僅僅是調整結構。

#### 手動驗證 (Manual Verification)
- **覆蓋漏洞**：針對重構影響的邏輯區塊，確保手動操作涵蓋了邊界條件與異常流程。
- **Agent 自檢**：Agent 應利用其分析能力，檢查重構後的程式碼邏輯是否與原始邏輯在功能上完全等價。

### 3. 執行流程 (Workflow)

重構任務應分為以下階段：

1. **[分析階段]**：閱讀 `code-review` 報告，計畫修改路徑。
2. **[場景設計階段]**：與使用者討論並列出本次重構需手動驗證的 Checklist。
3. **[執行階段]**：
    - 運用重構技術（如：Extract Method, Rename Variable, Replace Conditionals）。
    - 修正 `code-review` 指出的具體錯誤。
4. **[驗證階段]**：邀請使用者共同測試，或 Agent 根據預設場景進行邏輯自檢。

### 4. 輸出與說明

在完成重構後，Agent 應說明：
- **修改摘要**：對應了哪些審查建議。
- **驗證過程**：進行了哪些手動測試步奏。
- **待確認事項**：是否有需要使用者額外注意或在運行時觀察的部分。
