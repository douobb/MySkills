# Start Refactor Skill

將使用者要求或已驗證的 `code-review` finding 轉化為小步且可驗證的程式碼改善。純重構必須保持外部行為；邏輯或安全修復若會改變行為，必須明確列為修復並確認仍在使用者授權範圍內。

## When to use this skill

- 當需要依據先前 Code Review 的建議進行修復時。
- 當程式碼結構變得臃腫、難以維護，需要進行系統性優化時。
- 在修復 Bug 或增加新功能前，需要先進行預防性重構時。

## How to use it

### 1. 重構心態 (Mindset)

- **安全第一**：重構不應改變既有的外部行為。
- **漸進式修改**：小步快跑 (Small Steps)。每一次修改都應該是可驗證的。
- **驗證先行**：修改前先辨識現有測試、靜態檢查、建置與必要手動場景；只有缺少會實質改變方案的資訊時才詢問。
- **自動化優先**：優先執行既有且相關的自動化驗證，並在範圍允許且風險需要時補充或更新測試；手動驗證只補足無法可靠自動化的行為。
- **驗證 finding**：先對照目前程式碼、需求與測試確認審查建議仍成立，不盲目套用過時、範圍外或純偏好的建議。
- **保護現況**：工作樹不必乾淨；辨識並保留使用者既有修改。只有目標檔案存在無法安全隔離的重疊變更時才停止並請使用者決定。

### 2. 修復與改進原則 (Guidelines)

#### 遵循審查建議
- **對應修復**：找出 `code-review` 輸出中的「詳細說明」，逐項對應修改。
- **優先順序**：優先修復「嚴重程度：高」的問題（如安全性漏洞或邏輯錯誤）。

#### 安全重構步驟
1. **環境確認**：確認技術棧、適用規範、Git 差異、現有測試及目標範圍，保留無關與既有修改。
2. **建立基準**：安全可行時先執行最小相關驗證；既有失敗需記錄，不能歸因於本次修改。
3. **原子化修改**：一次只完成一個可獨立理解與驗證的改進，避免順手擴大到無關翻修。
4. **逐步驗證**：每個有風險的小步後執行受影響檢查；狀態未改變時不重複相同驗證。
5. **區分行為變更**：漏洞或邏輯修復若改變可觀察行為，分開說明需求依據、測試與相容性影響；超出授權時先停止。

#### 驗證策略 (Verification)
- **自動化檢查**：依變更執行既有測試、lint、type check、build 或 Skill 驗證；不得安裝未授權工具或以跳過檢查換取通過。
- **行為保護**：純重構以既有測試與新補充的回歸測試證明外部行為等價；無法完整驗證時回報限制，不宣稱完全等價。
- **手動場景**：UI、硬體、外部服務或其他難以自動化的路徑，提供精確步驟與預期結果，並區分已執行和建議使用者執行。

### 3. 執行流程 (Workflow)

重構任務應分為以下階段：

1. **[分析階段]**：讀取使用者要求、適用 finding、程式碼、測試與 Git 差異，界定不變行為及已授權修復。
2. **[基準階段]**：建立相關自動化與必要手動驗證基準，記錄既有失敗。
3. **[執行階段]**：小步套用合適重構技術；行為修復與純結構調整保持可辨識。
4. **[驗證階段]**：執行受影響的測試、靜態檢查與建置，必要時提供手動場景。
5. **[同步階段]**：進度或下一步改變時使用 `$write-project-todo`；已驗證的公開行為、指令或結構改變時使用 `$write-project-readme`。只有產品需求改變且使用者要求時才使用 `$write-project-prd`。 若同步 Skill 因私有文件追蹤、衝突或缺少授權而停止，保留已完成修改並將文件未同步列為交付阻塞，不得繞過。

### 4. 輸出與說明

在完成重構後，Agent 應說明：
- **修改摘要**：對應了哪些審查建議。
- **驗證過程**：列出已成功、失敗及未執行的自動化與手動檢查，不把建議場景說成已測試。
- **文件同步**：說明 TODO／README／PRD 是否需要及是否已同步。
- **待確認事項**：列出需要使用者額外觀察、決定或在真實環境驗證的部分。

不得因重構自動建立交接、暫存、提交或推送；只有切換 session 或使用者要求 Git 交付時，才使用對應 Skill。
