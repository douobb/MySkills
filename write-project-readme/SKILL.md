# 撰寫專案 README

建立或增量更新 README.md 與 README.en.md。README.md 是繁體中文語意來源；README.en.md 是對應英文版本。即使其中一份或兩份不存在，仍可執行；docs/PRD.md 不是必要條件。

## 核心原則

1. **先查證再撰寫**：有儲存庫內容可檢查時，不得只根據專案名稱或一句需求產生通用 README。
2. **只寫已驗證事實**：具體敘述必須由程式碼、設定、套件描述檔、建置腳本、測試、正式文件、發布資料、使用者指示或仍正確的既有 README 支持。
3. **區分實作與規劃**：以可執行程式碼及設定判斷已實作行為；只存在 PRD 的內容須標示為「規劃中」或省略，不得寫成已完成。
4. **服務新讀者**：優先說明專案用途、解決的問題、目前能力、安裝或執行方式，以及其他文件入口。
5. **保持雙語同步**：兩份 README 的事實、狀態、指令、版本、連結、引用、授權、發布資訊及警告必須等價；不得只更新單一版本。
6. **保留有效人工內容**：增量更新時保留仍正確的品牌、排版、警告、致謝與使用說明，只修正受變更影響或妨礙清晰度的內容。

無法驗證的資訊應省略；只有必要且無法安全省略時才標示「尚未確認」或詢問使用者。不得加入空白章節或 TODO、Coming soon 等預留文字。

## 儲存庫檢查

### 檢查來源

先完整讀取兩份既有 README，再依專案內容檢查：

- 原始碼、測試、docs/ 與 docs/PRD.md
- 套件描述檔、lockfile、runtime/toolchain 版本檔及建置腳本
- Container、CI、Release workflow、發布設定與變更紀錄
- .env.example、設定 Schema 與公開設定文件
- LICENSE、COPYING、NOTICE、CITATION.cff、AUTHORS、致謝或貢獻文件
- 適當且可用時的 Git remote、本機 tags、fork 中繼資料與原始碼授權標頭

偵測技術棧後，也要檢查其慣用的同類檔案。未經允許不得存取私人遠端資源。

建立內部事實清單，區分已驗證事實、已實作與規劃功能、不確定或過時內容、敏感資訊、來源公開狀態、技術版本、Release 狀態、授權及上游關係。

### 私有來源邊界

- 將 `TODO.md`、`docs/PRD.md`、`docs/private/` 及其他被忽略或未發布文件視為私有輸入，不因可以讀取就推定可公開。
- 私有輸入可協助理解規劃與背景，但不得在 README 直接連結、逐段摘要或複製內部 TASK／FR／Phase、驗收矩陣、排程、商業考量及交接內容。
- 只有內容已由程式碼、測試、設定、正式發布資料或其他公開來源獨立驗證，且移除私有脈絡後仍對讀者有價值時，才建立最小公開投影。
- 使用者明確要求公開 PRD、TODO 或私有文件時，先檢查 Git 追蹤狀態與敏感內容；需要整理時使用 `$manage-project-docs`，不得以 README 繞過其公開准入門檻。

### 衝突優先序

一般事實依序採信：

1. 目前可執行的程式碼與設定
2. 套件描述檔與建置腳本
3. 測試
4. 使用者目前的明確指示
5. 正式文件
6. 既有 README
7. PRD 規劃

技術版本依序採信：runtime/toolchain 版本檔、套件限制、lockfile、Container/CI、建置腳本、既有文件。不得只根據本機環境或一般生態知識推斷版本，不得使用無依據的 latest，也不得改變已驗證版本範圍的精度。

上游與引用關係依序採信：fork/remote 資訊、原始碼或授權標頭、NOTICE／CITATION.cff、使用者指示、既有致謝、正式文件。不得僅因相依套件或結構相似便宣稱衍生、所有權或抄襲。

### 何時詢問

僅在資訊不可判斷且會實質阻止正確 README 時詢問，例如：

- 專案用途或對外名稱不明
- 多個啟動指令互相衝突且無法驗證
- 必要的私人設定流程缺失
- 文件與實作有重大衝突
- 可能存在重要上游關係但無法確認

徽章、Emoji、截圖等裝飾性選項預設不必詢問；次要資訊缺少時直接省略相關章節。

## README 結構

每份 README 至少包含：

1. 已驗證的專案標題
2. 說明用途、目標情境與主要價值的簡短描述
3. 另一語言版本的相對連結

README.md 使用 [English](README.en.md)；README.en.md 使用 [繁體中文](README.md)，位置與格式應一致。

只加入有證據且對讀者有用的章節：

| English | 繁體中文 | 加入條件與要點 |
|---|---|---|
| Overview | 專案簡介 | 短描述不足，或需交代背景、使用者、情境、目標及重要上游關係時 |
| Features | 功能特色 | 只列已驗證能力；描述行為而非內部實作，未完成項目須標示狀態 |
| Demo | 展示 | 僅限已存在且權利明確的展示、截圖、GIF 或影片；儲存庫資源優先用相對路徑並檢查大小寫 |
| Requirements | 系統需求 | 已驗證的 OS、runtime、工具、外部服務或硬體條件；必要時區分執行與開發需求 |
| Technology Stack | 使用技術 | 只列有助理解、執行或開發的核心技術；保留可驗證版本及範圍，不列所有間接相依套件 |
| Installation | 安裝 | 可重現且順序正確；使用 lockfile 對應套件管理器、實際 scripts、路徑與必要建置／設定步驟 |
| Configuration | 設定 | 只記錄由範例、Schema、程式碼或正式文件支持的設定；標示必要性並以預留值表示機密 |
| Usage | 使用方式 | 提供符合實際行為的 CLI、UI、API 或函式庫範例；詳細內容連結至 docs/ |
| Development | 開發 | 僅記錄與一般使用不同且已驗證的開發、建置、lint、格式化、除錯或本機服務流程 |
| Project Structure | 專案結構 | 確實有助理解時，只列重要路徑，不列相依套件、輸出、快取或細碎檔案 |
| Testing | 測試 | 實際存在測試或測試指令時；不得推測覆蓋率或完整性 |
| Documentation | 文件 | 只連結存在且適合公開的文件；不得連結 `TODO.md`、預設私有的 `docs/PRD.md`、`docs/private/` 或其他被忽略／未發布規劃 |
| Release | 發布版本 | 有 tags、Release、changelog、發布設定、產物或 workflow 等正式證據時 |
| Roadmap | 開發藍圖 | 使用者要求或存在適合公開的高階規劃時；不得複製完整 TODO.md 或私人追蹤資訊 |
| References | 參考與致謝 | 存在已驗證的參考、改作、移植、fork、素材、模型、資料集、論文或其他來源時 |
| Contributing | 貢獻 | 有正式文件、模板、流程或使用者指示時；不得虛構 branch、commit、PR 或 review 規則 |
| License | 授權 | 有授權檔、套件中繼資料或使用者指示可驗證時；否則通常省略 |

標題與短描述已足夠時，不必另建「專案簡介」。架構、API、資料庫與部署細節通常留在 docs/，README 只保留必要摘要與入口。徽章不是獨立章節；僅在目標和值均可驗證且有用時精簡加入。

## 關鍵內容規則

### 指令、版本與設定

- 安裝、啟動、建置、測試及開發指令必須取自實際檔案並保持正確順序；例如存在 pnpm-lock.yaml 時，不得無依據改寫為 npm install。
- 版本必須保留來源支持的精度，例如主版本、最低版本、精確鎖定版本或支援範圍；無法判斷時省略或說明未明確鎖定。
- 不得虛構支援平台、相容性、API 端點、輸出、效能或安全保證。
- 不得將真實 .env 值、token、密碼、private key、私人路徑或端點寫入 README；機密範例使用 API_KEY=your_api_key 等預留值。

### 引用、授權與發布

若專案實質 fork、衍生或改作自可識別的儲存庫，必須：

1. 在 Overview／專案簡介前段簡短、直接連結並準確揭露關係。
2. 引導讀者到 References／參考與致謝。
3. 在 References 記錄來源名稱、連結、關係性質、主要修改及相關授權或引用要求。

關係用詞必須精確：Based on 表示大量實作或結構衍生；Forked from 表示實際 fork 或延續；Adapted from 表示修改部分程式碼、設計或概念；Inspired by 表示實作大致獨立；Uses 表示作為函式庫、工具、素材、資料集或服務。不得弱化大量改作關係，也不得在兩種語言使用不同強度的描述。

授權不明時不得宣稱 MIT、Apache、GPL、開源或專有，也不得自動選擇或建立授權。引用要求不明時說明需進一步授權審查，不要猜測。

Release 資訊只可來自可驗證的版本、日期、連結、產物與狀態；不得虛構最新版本、下載連結、自動更新能力，不得把預設分支或 pre-release 描述為穩定版。

## 寫作與同步

- 中文版使用台灣自然繁體中文；指令、路徑、識別字與程式碼保持原樣，避免空泛宣傳。
- 英文版使用自然技術英文，不做逐字機械翻譯。
- 兩份文件章節涵蓋與事實必須等價；命令、版本、URL、檔名及 Release identifier 通常完全相同。
- 若單一版本包含額外且正確的資訊，先驗證，再同步到兩份文件。
- 既有 README 採增量更新：辨識正確、過時、無依據、重複及單語內容；移除失效連結與過時指令，但不要因不符合模板便刪除有效自訂內容。

## 工作流程

1. **檢查儲存庫**：讀取相關程式碼、設定、測試、文件、引用、發布資料及兩份 README。
2. **建立事實清單**：區分已驗證、規劃中、不確定、過時、敏感及私有來源資訊。
3. **選擇章節**：只保留有證據且有讀者價值的內容。
4. **撰寫中文版本**：以 README.md 建立共同語意。
5. **同步英文版本**：依中文內容與同一事實清單更新 README.en.md，不得加入英文獨有資訊。
6. **驗證內容**：檢查或安全執行指令，確認路徑、檔名、連結、設定、版本、引用、Release 與下載資訊；不得修改專案來迎合 README。
7. **比較雙語**：確認章節、事實、狀態、警告與措辭強度一致。
8. **回報結果**：摘要檔案與主要章節變更、因缺乏證據而省略的資訊、重要未決事項，以及授權、上游、Release 和指令實測狀態。

只有實際成功執行的指令才能回報為「已測試」。

## 完成檢查

完成前確認：

- 兩份 README 都存在、互相連結，且分別使用自然繁體中文與英文。
- 專案名稱、描述、章節、功能及狀態語意同步。
- 所有具體敘述有證據；規劃內容未被描述為已完成。
- 指令、路徑、檔名、相對連結、設定與版本正確且一致。
- 沒有機密、空白章節、預留文字、失效連結或無依據的功能、相容性、授權及 Release 宣稱。
- 沒有連結或洩漏預設私有的 TODO、PRD、`docs/private/` 與內部追蹤脈絡；公開投影可由公開證據獨立理解。
- 已適當保留有效人工內容、引用、授權聲明及重要上游關係。
- 實質上游關係已在 Overview 前段揭露，並於 References 詳述。
- Release、License、Contributing、Testing、Demo、Badge 等選用內容均有證據。
- Markdown 標題、表格與程式碼區塊有效。

不得刪除無關檔案、為配合文件修改原始碼，或在未明確要求時暫存、提交、推送變更。

只有在完成上述檢查並回報驗證範圍後，才算完成。
