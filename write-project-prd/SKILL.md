# 撰寫專案 PRD

將已知需求整理成清楚、可驗證且適合後續開發的 docs/PRD.md。描述「要做什麼」與「如何判斷完成」，不要取代 TODO.md、架構／API／資料庫設計或實作工作。

## 需求來源與資訊缺口

先利用現有上下文，不要重複詢問已知資訊。依專案情況檢查：

- 使用者目前及先前提供的需求
- 既有 docs/PRD.md、中英文 README 與 docs/ 文件
- 原始碼、設定、測試、Issue、規格、筆記及使用者指定來源

來源衝突時：

1. 優先採用使用者目前的明確指示。
2. 保留仍有效的既有需求。
3. 將無法判定的衝突列入「待確認事項」。
4. 不要默默選擇其中一方或虛構結論。

只有缺少會實質影響 PRD 的核心資訊時才詢問，優先確認問題、目標使用者、核心功能、範圍、非目標、關鍵限制及完成標準。一次最多詢問 3–5 個高價值問題；若已足以整理初稿，將剩餘缺口列入「待確認事項」，不要加入假設性需求。

## 內容原則

- 將 `docs/PRD.md` 視為預設不公開的本機產品規劃；可記錄必要的內部產品脈絡，但不得保存憑證、完整敏感值或不必要個資。
- 描述目標產品與可觀察結果，不要撰寫程式碼或實作說明。
- 將需求寫得具體、可理解、可驗證，並區分已確認需求、規劃方向與待確認事項。
- 不要把建議、推測或技術偏好偽裝成正式產品需求。
- 可記錄重大技術限制，但將詳細設計留在其他文件。
- 不要加入工時、任務勾選框、開發階段或逐檔案步驟。
- 不要因尚未實作就刪除合法需求；只有文件明確追蹤狀態且有證據時，才聲稱需求已實作。
- 依專案規模選擇內容，不要強迫小型專案加入商業 KPI、完整 Persona 或其他無關章節。
- 不要留下 TODO、TBD、模板文字、空白章節、未確認數值或憑證等機密。

## 隱私與 Git 邊界

- 寫入前檢查專案根目錄 `.gitignore`、PRD 的追蹤／暫存狀態，以及可用時的 `git check-ignore`。
- 除非使用者明確要求追蹤或上傳這一份 PRD，根目錄 `.gitignore` 必須含獨立規則 `docs/PRD.md`；缺少時保留既有內容並增量補上，不得重複。
- 若 PRD 已被 tracked、staged 或存在於待推送 commit，而目前沒有明確公開意圖，先停止並回報；不得自行 untrack、改寫歷史或繼續產生可能被意外上傳的修改。
- 明確要求公開 PRD 只代表可評估追蹤，不代表所有內容都適合公開；先檢查商業規劃、個資、安全資訊與引用，再由使用者決定保留完整 PRD 或建立去敏的公開版本。

## 文件結構

依需要選用下列章節：

```markdown
# 產品需求文件
## 文件資訊
## 專案概述
## 問題與背景
## 目標
## 非目標
## 目標使用者
## 使用情境
## 功能需求
## 非功能需求
## UX 與互動需求
## 資料與整合需求
## 限制與相依性
## 驗收條件
## 成功指標
## 風險
## 待確認事項
## 參考來源
```

至少涵蓋專案概述、問題或目標、功能需求、範圍或非目標及驗收條件；只有存在資訊缺口時才加入「待確認事項」。

### 功能需求

為核心功能使用穩定 ID：

```markdown
### FR-001：功能名稱

**說明：** 功能要達成的結果。

**驗收條件：**

- 可觀察且可驗證的結果。
- 失敗或邊界情況的預期行為。
```

- 每項需求只描述一個主要能力，驗收條件只寫結果，不指定實作方法。
- 建立 ID 後盡量保持穩定；刪除需求時不要重複使用舊 ID。
- 只有需要時才加入 Must／Should／Could 等優先級。

### 非功能需求與引用

只記錄已知的效能、安全、隱私、可用性、相容性、可維護性、可存取性、離線、資料保存或備份要求；不要自行虛構數值目標。

若 PRD 參考、改作或以其他專案、儲存庫、論文、教學或規格為基礎，在「參考來源」記錄名稱、連結或來源、關係性質及採用或修改範圍。不要將一般相依套件自動視為參考來源。

## 建立與更新流程

1. 檢查需求來源及既有 PRD，整理已確認、衝突、缺失與假設資訊。
2. 必要時詢問少量關鍵問題，否則將缺口列為待確認事項。
3. 選擇適用章節並撰寫或增量更新 docs/PRD.md。
4. 檢查目標、範圍、需求 ID、引用與驗收條件。
5. 回報主要變更、仍待確認事項與驗證範圍。

首次建立時，從已知需求產生完整可用初版，不要開始實作。

更新既有 PRD 時：

- 先完整讀取原文，找出新增、修改、移除與衝突需求。
- 保留仍有效的人工內容與需求 ID，只更新受影響章節。
- 只有整體重整能明顯改善一致性時才擴大修改，不要無必要重寫全文。
- 將已取消範圍移至「非目標」，或明確移除並在完成回報中說明。

## 完成檢查

完成前確認：

- docs/PRD.md 存在且非空白，結構清楚並只含適用章節。
- 需求與目標一致，範圍、非目標、限制及相依性明確。
- 核心功能有穩定 ID 與可觀察、可驗證的驗收條件。
- 已確認、規劃及待確認資訊清楚分開，沒有明顯矛盾或虛構事實。
- 沒有模板文字、空白章節、實作任務、程式碼或不必要技術設計。
- 既有 PRD 採增量更新，參考來源及必要引用已正確保留。
- 未刪除無關檔案、揭露機密，或在未明確要求時暫存、提交、推送變更。
- PRD 已保持忽略，或使用者已明確確認其公開追蹤；任何 `.gitignore` 變更與既有追蹤風險均已回報。

完成回報應說明已建立或更新的檔案、主要需求變更、需求 ID 是否保留、未決衝突或待確認事項、參考來源，以及未開始程式實作。

只有完成上述檢查並回報結果後，才算完成。
