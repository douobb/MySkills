# Code Review Skill

本 Skill 旨在提供專業、客觀且具建設性的程式碼審查意見。Agent 將扮演資深開發者的角色，指出潛在問題並提供改進建議，但**嚴禁主動修改原始碼**。

## When to use this skill

- 當使用者要求對特定的程式碼片段、檔案或 Pull Request 進行審查時。
- 當需要評估程式碼是否符合最佳實踐、安全性要求或效能標準時。
- 在大型重構或新功能實作完成後，進行最後的品質檢查。

## How to use it

### 0. 確認範圍與證據

- 先確認審查目標是工作樹差異、指定 commit／PR、特定檔案或完整模組，不把範圍外既有問題混入本次結論。
- 讀取適用需求、測試、設定、型別、呼叫端與專案規範；工作樹不必乾淨，但不得覆寫或修改任何內容。
- 區分本次引入、既有但相關、無法驗證與純風格偏好；沒有可重現路徑或程式證據時降低確信度，不把猜測寫成缺陷。
- 可執行唯讀或不改變專案狀態的檢查；不得為審查安裝套件、改設定、產生修復檔案或變更 Git 狀態。

### 1. 審查心態 (Mindset)

- **客觀公平**：評論應針對程式碼而非作者。
- **建設性意見**：指出問題的同時，應說明「為什麼」這是一個問題，並提供具體的改進方向。
- **專注品質**：平衡開發速度與長期可維護性。
- **僅限審查**：**不要**直接應用修復。所有的建議都應以文字說明或程式碼範例的形式呈現。

### 2. 必查項目 (Checklist)

#### 正確性 (Correctness)
- **邏輯漏洞**：程式邏輯是否符合需求？是否存在潛在的 Bug？
- **邊界條件**：是否處理了空值、極值、或是無效的輸入？
- **異常處理**：異常捕捉是否過於籠統（如捕捉所有 Exception）？錯誤訊息是否具備資訊量？

#### 可讀性與可維護性 (Readability & Maintainability)
- **語意命名**：變數、函式與類別的命名是否清晰易懂？
- **複雜度控制**：函式是否過長？巢狀層級是否過深？
- **重複代碼 (DRY)**：是否存在可以提取為共用元件的重複邏輯？
- **註解品質**：註解是否說明了「為什麼」這樣做，而非僅僅重複程式碼內容？

#### 架構與設計 (Architecture & Design)
- **模組化**：是否符合單一職責原則 (SRP)？
- **耦合度**：元件之間是否過度依賴？
- **擴展性**：設計是否考慮到未來的擴充性？是否過度設計 (Over-engineering)？

#### 效能 (Performance)
- **複雜度分析**：演算法的時間與空間複雜度是否合理？
- **資源管理**：是否有資源洩漏（如檔案或網路連線未關閉）？
- **重複運算**：在迴圈中是否有不必要的重覆計算或 I/O 操作？

#### 安全性 (Security)
- **輸入驗證**：是否防範了 SQL 注入、XSS 等常見攻擊？
- **敏感資訊**：程式碼中是否包含 API Key、密碼或其他敏感資訊？
- **授權檢查**：敏感操作是否具備適當的權限檢查？

#### 風格與規範 (Style & Standards)
- **語言慣例**：是否符合該程式語言的 Idiomatic 寫法（如 Python 的 PEP8，JavaScript 的 Airbnb 風格）？
- **專案一致性**：編碼風格是否與專案中現有的程式碼保持一致？

### 3. 特定語言指引 (Language-Specific Guidance)

#### Dart / Flutter 專案
- **參考資料**：只有目標包含 Dart／Flutter 時才完整讀取 `references/codereview-flutter.md`。
- **適用性**：該清單是條件式檢查訊號，不是通用架構或套件規範；只套用可由專案設定、既有慣例與需求支持的項目。
- **優先順序**：以目前使用者要求、專案規範與可驗證行為為準，不因參考清單而要求無關重構、套件升級或架構替換。
- **重點關注**：Widget 拆分、State 管理、異步操作 (Future/Stream) 的安全性。

### 4. 嚴重程度

- **高**：可能造成安全事件、資料遺失、核心功能錯誤或無安全替代方案的發布阻塞。
- **中**：在合理情境下造成錯誤、相容性或顯著維護風險，但有受限影響或替代方案。
- **低**：確實可改善且有具體影響的局部問題；純偏好不列為 finding。

### 5. 輸出格式 (Output Format)

請依照以下結構回報審查結果。每項 finding 必須包含可定位位置、觸發條件、影響與建議；無法確認時明列假設。若發現**多個問題**，請確保條理清晰並按嚴重程度或類別分組：

- **(A) 總體評價**：簡述程式碼的整體品質、優點及主要待改進之處。
- **(B) 問題清單與建議**：
    - **摘要表格**（若問題超過 2 個時建議提供）：
        | 嚴重程度 | 類別 | 描述摘要 | 建議行動 |
        | :--- | :--- | :--- | :--- |
        | [高/中/低] | [例如：安全性] | 簡短描述問題 | 簡短建議 |
    - **詳細說明**（逐項深入探討）：
        1. **[類別] - 問題名稱**：
            - **嚴重程度**：高/中/低。
            - **具體位址**：指明檔案名稱與大致行號。
            - **問題描述**：詳細說明為何此處存在問題。
            - **改進建議**：提供具體的調整思路。
            - **程式碼範例**：
              ```diff
              - 原始程式碼
              + 建議修改方向
              ```
- **(C) 總結**：給出整體的通過建議（例如：✅ 建議核准、⚠️ 建議微調後通過、❌ 建議修正後重新審查）。若沒有具體 finding，明確寫「未發現需回報問題」，並列出未驗證範圍與殘餘風險。

審查完成後不得自動呼叫重構、文件或 Git Skill；只有使用者要求修正或交付時，才由對應 Skill 接手。
