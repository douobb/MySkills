# 管理專案 Git

統一首次初始化與日常提交上傳流程。以保留使用者工作、避免資料外洩及維持可追溯性為優先；不得為了完成推送而繞過驗證、覆蓋遠端或擅自修改歷史。

## 核心原則

- 先確認使用者要求的是「只初始化」、「只提交」、「只推送」或「提交並推送」，不得擴大外部變更範圍。
- 只處理本次要求涉及的儲存庫與檔案；保留無關修改、既有分支、tag、remote 與暫存狀態。
- 先檢查再暫存，先驗證暫存內容再提交，先比較遠端版本再推送。
- 使用精確路徑暫存；只有逐檔審查整個工作樹後才可使用廣泛加入方式。
- 不得自動執行 `reset --hard`、`clean`、捨棄修改、刪除分支、重寫歷史、`commit --amend` 或強制推送。
- 一般提交與非強制推送可在使用者已明確要求時，通過全部檢查後連續完成；發現差異、風險或錯誤時必須停止並交由使用者選擇。
- 敏感資訊掃描只能降低風險，不得宣稱保證不存在機密。
- 不輸出完整 token、密碼、私鑰或其他機密值；只回報檔案、類型與必要的遮罩位置。

## 選擇流程

1. 若目前目錄不是 Git 儲存庫，或使用者明確要求第一次初始化，執行「流程 A」。
2. 若已有 `.git/` 或可解析 Git 根目錄，執行「流程 B」。
3. 若偵測到巢狀儲存庫、submodule、worktree 或目前路徑與使用者指定專案不一致，先回報實際根目錄與影響範圍；無法確定目標時停止。
4. 若使用者只要求檢查或準備提交，完成提交前報告後停止，不得自行提交或推送。

## 共用提交前檢查

### 1. 盤點儲存庫與工作樹

確認並記錄：

- Git 根目錄、目前分支、HEAD、工作樹狀態及是否為 detached HEAD。
- staged、unstaged、untracked 與刪除／重新命名檔案；必要時辨識 ignored 檔案。
- remote 名稱與 URL、upstream、目前分支相對遠端的 ahead／behind 狀態。
- commit 前確認有效的 `user.name`、`user.email` 與既有簽署規範；缺少或可能不適合公開時停止，不得擅自修改 global 設定，應說明 repository-local、global 與隱私信箱選項。
- 是否正在 merge、rebase、cherry-pick、revert 或 bisect，以及是否有未解決衝突。
- 既有 staged 內容是否屬於本次範圍。若索引混有使用者先前暫存且不應進入本次提交的檔案，停止並請使用者決定，不得擅自取消暫存或一併提交。

先讀取 `AGENTS.md`、貢獻指南及儲存庫內的 Git／測試規範。不得假設預設分支、remote 名稱或 commit 格式。

### 2. 檢查敏感資訊與多餘檔案

逐一檢查預計提交的 tracked 變更與 untracked 檔案；若包含推送，亦須檢查尚未存在於目標遠端的完整 outgoing commit 範圍。至少涵蓋：

- `.env`、憑證、私鑰、公私鑰容器、雲端認證、cookie、session、資料庫連線字串及含帳密 URL。
- API key、access token、password、secret、authorization header 等高風險內容；優先使用儲存庫既有的 secret scanner，沒有時再以檔名、內容模式與人工語意檢查補足，不得自動安裝工具。
- 個人資料、客戶資料、內部端點、商業規劃、未公開 PRD／TODO、`docs/private/` 及其他不應公開內容。
- log、除錯輸出、快取、coverage、build 產物、套件目錄、暫存檔、備份、資料匯出、資料庫、壓縮檔、過大二進位檔、IDE／OS 中繼資料與無閱讀價值的紀錄文件。
- `.gitignore` 是否涵蓋確認不應追蹤的項目，且規則沒有誤排除必要來源檔。

若發現疑似機密：

1. 立即停止暫存、提交與推送。
2. 不顯示完整值，只回報檔案與風險類型。
3. 若尚未提交，建議移除內容、改用環境變數／機密管理並補充忽略規則。
4. 若已進入任何 commit 或遠端，說明 `.gitignore` 無法清除歷史，優先建議撤銷或輪替憑證，再提出歷史清理選項；未經明確同意不得改寫歷史或強制推送。

多餘檔案不得直接刪除。列出精確路徑、判定理由，以及「忽略、保留、移除或改放其他位置」等方案，由使用者決定。

### 3. 同步程式碼、規劃與文件

比較程式碼、設定、測試及公開行為與文件：

- 已驗證的功能、指令、設定、限制、版本或專案結構改變時，使用可用的 `$write-project-readme` 同步 `README.md` 與 `README.en.md`。
- `docs/` 含開發紀錄、重複文件、私有規劃或不適合公開的內容時，使用 `$manage-project-docs` 先盤點；該 Skill 要求確認時，暫停 Git 流程等待使用者決定。
- 專案以本機 `TODO.md` 維護進度且實作狀態已改變時，使用 `$write-project-todo` 更新；保持其 Git 忽略狀態，不因同步而自動上傳。
- 只有產品需求本身確實改變且使用者要更新需求時，才使用 `$write-project-prd`；PRD 預設維持私有與忽略。
- 若空白目錄同時需要專案骨架，可先使用 `$initialize-project-structure`；它不得自行初始化 Git，完成後再回到流程 A。
- 若程式碼變更範圍大、風險高或使用者要求交付前審查，可使用 `$code-review` 先產生唯讀審查結果；不得因審查結果自動擴大修正範圍。

其他 Skill 不是固定依賴。僅在可用且符合觸發條件時使用；任何 Skill 修改檔案後，都必須重新執行工作樹、敏感資訊、文件與驗證檢查。

### 4. 執行品質驗證

- 依儲存庫既有指令執行適合本次變更的測試、lint、格式、建置、型別檢查或 Skill 驗證。
- 執行 `git diff --check`，並檢查 staged diff 的內容與範圍。
- 不得自行安裝相依套件、變更工具鏈或修改程式碼來掩蓋驗證失敗。
- 無測試、無法執行或驗證失敗時，清楚回報「未驗證」或失敗證據。失敗若可能影響提交品質，停止並由使用者決定修正、縮小範圍或接受風險。

## 流程 A：第一次初始化

### 1. 驗證初始化條件

- 檢查所有一般與隱藏項目，確認目標目錄及是否已有 Git metadata。
- 若已是 Git 儲存庫，不得再次初始化；切換到流程 B。
- 初始化 Git 不等於建立專案架構、選擇授權或建立遠端儲存庫。只有使用者同時要求時才執行這些工作。
- 若使用者未指定初始分支，預設使用 `main` 並在結果中說明。

### 2. 準備追蹤邊界

- 在 `git init` 前先完成共用敏感資訊與多餘檔案檢查。
- 檢查或建立符合目前專案的 `.gitignore`；不得用通用模板覆寫既有規則。
- 預設排除本機 `TODO.md`、`docs/PRD.md`、`docs/private/`、真實 `.env` 與已確認的產生物；若使用者明確要求公開 PRD，重新確認內容適合公開後才追蹤。
- 不得自動選擇或建立 `LICENSE`。

### 3. 初始化與設定遠端

1. 以 `git init -b <branch>` 初始化；若 Git 版本不支援，使用等價且不破壞內容的方式設定初始分支。
2. 驗證 Git 根目錄與目前分支。
3. 只有使用者提供或確認 remote URL 時才新增 remote。預設名稱可使用 `origin`；若名稱或 URL 已存在且不同，停止並回報。
4. 若要求建立 GitHub／GitLab 等遠端儲存庫，先取得平台、名稱、擁有者與 public／private 選擇；建立遠端屬於獨立外部操作，不得從本機資料自行推定公開性。

若只要求初始化，到此回報並停止，不建立 commit 或推送。

### 4. 初始提交與首次推送

使用者同時要求提交或上傳時：

1. 執行完整共用提交前檢查與品質驗證。
2. 列出預計提交與排除檔案，使用精確路徑暫存。
3. 檢查 `git diff --cached --check`、完整 staged diff 與檔案統計。
4. 根據實際內容產生簡潔 commit message；不得宣稱未完成或未驗證的功能。
5. 建立 commit，記錄 commit hash。
6. 推送前先查詢遠端分支。遠端不存在同名分支時，可使用一般 `git push -u <remote> <branch>`。
7. 若遠端已有歷史而本機無共同基底，停止並提出：檢查遠端內容後整合、改用新分支／PR，或在充分理解風險後取代遠端等方案；不得自行使用 `--force` 或合併不相關歷史。

## 流程 B：既有儲存庫提交與上傳

### 1. 確認提交範圍

- 從使用者要求、目前變更與既有工作脈絡建立本次檔案清單。
- 若變更同時包含多個無關目的，建議拆成原子提交；是否拆分由使用者決定。
- 若沒有實質變更，不建立空 commit，除非使用者明確要求且理解用途。

### 2. 比較遠端版本

推送前執行安全的遠端資訊更新，例如 `git fetch --prune`，再比較本機分支與 upstream／目標遠端分支：

- 本機領先且未落後：可繼續一般推送。
- 本機落後：停止，提出先檢查後 rebase、merge 或放棄本次推送等方案。
- 雙方分歧：停止，回報各自 commit 數與相關提交，提出 rebase、merge 或新分支／PR 方案。
- upstream 缺失：確認目標 remote 與 branch；使用者已明確要求上傳且遠端無同名分支時，才可設定 upstream。
- fetch、驗證、hook、認證、網路、LFS、分支保護或權限失敗：保留完整錯誤摘要與目前狀態，提出針對性方案，不得以停用 hook、跳過驗證或強制推送繞過。

不得自動 pull、merge 或 rebase。使用者選擇整合方案後，執行前重新確認工作樹安全；整合或解決衝突後必須重跑完整提交前檢查與驗證。發生衝突時停止，列出衝突檔與可行處置，不替使用者猜測內容取捨。

### 3. 暫存與提交

1. 完成共用檢查、必要文件同步與驗證。
2. 逐一使用精確路徑加入本次檔案，避免把無關 untracked 或 ignored 檔案帶入。
3. 再次審查 staged 檔案清單、內容、敏感資訊、檔案大小及 `git diff --cached --check`。
4. 依 staged diff 撰寫單一目的、可追溯的 commit message，遵循儲存庫既有格式；未規範時使用簡潔的繁體中文祈使或結果描述。
5. 建立 commit。不得自動 amend、簽署、建立 tag 或產生 release。

若 commit hook 失敗，回報 hook 名稱與錯誤。修正後必須重跑相關檢查，不得使用 `--no-verify`，除非使用者在理解風險後明確選擇。

### 4. 推送與驗證

1. 在 push 前重新確認 HEAD、branch、remote URL 與 ahead／behind 狀態，避免檢查後狀態已改變。
2. 使用一般 push；不得自動推送其他分支、tag 或所有 refs。
3. 非 fast-forward 時停止，不得自動 retry 為 force push。
4. 只有使用者明確指定要改寫遠端歷史、確認精確 remote／branch 並接受影響後，才可考慮 `--force-with-lease`；不得使用裸 `--force`。
5. 推送後比較本機 HEAD 與遠端追蹤分支，確認 commit hash 一致，並再次檢查工作樹是否仍有未提交內容。

## 異常回報格式

發現版本差異或錯誤時，先停止會擴大影響的動作，回報：

1. **目前狀態**：branch、HEAD、remote、ahead／behind、工作樹與已完成動作。
2. **問題證據**：精簡錯誤訊息、衝突檔案或風險類型，不揭露機密值。
3. **影響**：哪些提交、檔案、遠端或其他協作者可能受影響。
4. **建議方案**：列出安全選項、優缺點與推薦方案；包含「暫不處理」選項。
5. **等待決定**：未取得使用者選擇前，不執行 merge、rebase、取消追蹤、歷史清理、改 remote、刪除或強制推送。

## 完成回報

列出：

- 執行的是初始化、提交、推送或其組合。
- Git 根目錄、branch、remote 與 upstream。
- 新 commit message、短 hash 及實際納入的檔案。
- 敏感資訊與多餘檔案檢查範圍，以及任何排除項目；不得宣稱零風險。
- 文件同步情況與呼叫過的其他 Skill。
- 實際成功的測試與驗證；未執行或失敗項目須明示。
- push 結果、本機與遠端 commit 是否一致。
- 仍存在的 staged、unstaged、untracked、ignored 風險或待使用者處理事項。

只有所有已授權動作成功、遠端一致性已驗證，且未隱藏剩餘風險時，才可回報流程完成。
