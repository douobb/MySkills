# 撰寫專案 TODO

將 PRD 與目前專案狀態轉換成可執行、可排序、可驗證的 TODO.md，管理「接下來要做什麼」。不要取代 PRD、技術設計或實作工作，也不要自動將本機 TODO.md 加入版本控制。

## 資料來源與優先序

依專案情況檢查：

- docs/PRD.md 與既有 TODO.md
- 中英文 README、docs/、Issue 及變更紀錄
- 原始碼、測試、設定與套件描述檔
- 可安全取得的 Git 狀態與近期差異
- 使用者目前的明確指示

來源衝突時依序採信：

1. 使用者目前的明確指示
2. PRD 的有效需求
3. 目前程式碼、測試與設定所證明的狀態
4. 既有 TODO.md
5. 其他文件與歷史資訊

不要自行解決無法判定的衝突；將其列入「待確認事項」並回報。PRD 缺失或不足時，只能依可靠資訊建立有限規劃並明確列出缺口，不要虛構需求。

## 規劃原則

- 將 `TODO.md` 視為預設不公開的本機執行規劃；可保留必要的任務脈絡，但不得保存憑證、完整敏感值或不必要個資。
- 將 PRD 需求轉成實作工作，不要直接把 PRD 標題複製成勾選框。
- 讓每個任務都有明確成果及可觀察或可執行的完成方式。
- 依真實相依關係排序，優先安排能解除阻塞的基礎工作。
- 檢查相依關係是否形成循環；若存在循環，不要產生無法開始的排序，應調整拆分或列為阻塞事項。
- 視需要納入測試、文件、安全、整合與發布準備，不要只規劃功能程式碼。
- 不要撰寫程式碼、修改 docs/PRD.md、建立詳細架構／API／資料庫設計，或自動執行任何任務。
- 不要加入工時臆測、虛構進度、測試結果、機密、模板文字、空白階段或無意義任務。

## 隱私與 Git 邊界

- 寫入前檢查專案根目錄 `.gitignore`、TODO 的追蹤／暫存狀態，以及可用時的 `git check-ignore`。
- 除非使用者明確要求追蹤或上傳這一份 TODO，根目錄 `.gitignore` 必須含獨立規則 `TODO.md`；缺少時保留既有內容並增量補上，不得重複。
- 若 TODO 已被 tracked、staged 或存在於待推送 commit，而目前沒有明確公開意圖，先停止並回報；不得自行 untrack、改寫歷史或繼續產生可能被意外上傳的修改。
- 明確要求公開 TODO 時仍須先檢查內部 task／phase、商業規劃、個資及安全資訊；必要時建議改成不含內部追蹤細節的公開 roadmap。

## 任務與階段

### 任務粒度

讓每個任務能在一段合理、集中的開發工作中完成。若任務包含多個獨立成果、跨越多個系統層級或無法一次驗證，將其拆分；不要細到逐檔案、逐函式或逐次指令。

過大：

```markdown
- [ ] 完成使用者系統
```

適當：

```markdown
- [ ] 建立使用者資料模型與遷移
- [ ] 實作登入與登出流程
- [ ] 加入 session 驗證與過期處理
- [ ] 補充登入失敗與 session 過期測試
```

### 任務格式

小型任務可使用：

```markdown
- [ ] 任務名稱
```

核心或有相依關係的任務可使用：

```markdown
- [ ] TASK-001 任務名稱
  - 對應需求：FR-001
  - 依賴：TASK-000
  - 驗證：可觀察或可執行的完成檢查
```

- 每項任務只承擔一個主要成果；驗證不得只寫「確認正常」。
- 建立 TASK ID 後盡量保留，刪除任務時不要重複使用舊 ID。
- 只有存在真實相依關係時才標示依賴，不要為每個小任務加入完整中繼資料。

### 階段與文件結構

依相依順序安排具有明確目標與完成邊界的階段，例如基礎架構、核心資料與服務、主要功能、整合與錯誤處理、品質及發布準備。複雜階段可加入 checkpoint；小型專案可使用單層任務清單，不要強制固定階段。

依需要選用：

```markdown
# 專案實作規劃

## 目前狀態
- 目前階段：
- 下一步：
- 阻塞事項：

## 階段 1：名稱
**目標：**
- [ ] 任務

## 跨階段工作
## 待確認事項
## 已取消或延後項目
```

不要建立空白章節。只有能由可靠任務狀態計算時才加入進度百分比。

## 完成狀態

使用 [ ] 表示未完成、[x] 表示已完成。只有實作符合任務範圍，且相關測試、建置、文件、驗證或使用者確認提供可信證據時，才能標記 [x]；檔案存在本身不代表完成。

部分完成或無法驗證時保持 [ ]，並視需要記錄「狀態／已完成／尚缺」。

**TASK 完成不代表對應 FR 已驗收。** 只有 PRD 中該功能的所有適用驗收條件均有證據滿足時，才能回報 FR 已驗收；不要由單一任務狀態推斷需求整體完成。

## 建立與更新流程

1. 檢查 PRD、既有 TODO 及目前儲存庫，整理需求、已完成能力、缺口與阻塞事項。
2. 將未完成需求拆成適當粒度的任務，建立需求對應。
3. 安排依賴與階段，檢查循環依賴及無法開始的任務。
4. 加入必要品質工作，建立或增量更新 TODO.md。
5. 驗證任務狀態、需求對應、完成方式、下一步與阻塞事項。
6. 回報主要變更與建議下一步，不要開始實作。

首次建立時，依可靠資訊建立完整可用的初版。

更新既有 TODO 時：

- 先完整讀取原文，保留仍有效的階段、任務與 ID。
- 依 PRD、程式碼、測試與設定變更新增或調整任務。
- 只有取得證據時才更新完成狀態；將取消或延後工作移至對應章節。
- 更新目前階段、下一步與阻塞事項，不要無必要地重寫、重新編號或重排全文。

## 完成檢查

完成前確認：

- TODO.md 存在且非空白，任務能對應 PRD、目前缺口或必要品質工作。
- 任務粒度適中、成果明確，核心任務有可判定的完成方式。
- 階段與任務依相依關係排序，沒有未處理的循環依賴。
- 每個 [x] 都有可信證據，部分完成任務沒有被錯誤勾選。
- TASK 與 FR 狀態分開判定，未將單一任務完成誤報為需求已驗收。
- 下一步、阻塞、取消與延後事項符合目前狀態。
- 既有 TODO 採增量更新，沒有程式碼、未確認事實、機密、模板文字或空白章節。
- 未刪除無關檔案，或在未明確要求時暫存、提交、推送變更。
- TODO 已保持忽略，或使用者已明確確認其公開追蹤；任何 `.gitignore` 變更與既有追蹤風險均已回報。

完成回報應說明主要新增、修改、完成、取消或延後任務，是否保留 ID，目前階段、下一步、阻塞與待確認事項，並確認未開始程式實作。

只有完成上述檢查並回報結果後，才算完成。
